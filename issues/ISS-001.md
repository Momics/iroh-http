# [ISS-001] Server can panic/abort for small `maxHeaderBytes`

## Summary

The server passes `maxHeaderBytes` directly to hyper's `max_buf_size` without clamping it to a minimum of 8192. Values below that threshold cause a panic that in release builds triggers an abort.

## Evidence

- `crates/iroh-http-core/src/server.rs:573` — un-clamped `max_buf_size(max_header_size)` call
- `docs/internals/http-engine.md:164` — internal docs require clamp to `max(8192)` and warn that smaller values panic
- `docs/internals/design-decisions.md:69` — same clamp requirement documented
- `Cargo.toml:35` — release profile sets `panic = "abort"`, converting panics to process termination

## Impact

Runtime crash in production when a misconfigured or unusually small `maxHeaderBytes` value is provided. In release builds this aborts the whole process with no recovery path.

## Remediation

1. Change the server builder to `.max_buf_size(max_header_size.max(8192))`.
2. Add post-parse header byte measurement and an explicit reject path (see ISS-003).

## Acceptance criteria

1. `maxHeaderBytes = 1` no longer crashes the server.
2. Oversized headers return a deterministic error response with no panic.
