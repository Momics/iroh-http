# [ISS-001] fetch() signature is not web-standard — should accept httpi:// URL instead of (peer, path)

## Summary

`node.fetch()` currently takes `(peer: PublicKey | string, input: string | URL, init?)` — a separate peer ID and path. The web-standard `fetch()` takes `(input: string | URL | Request, init?)`. Users should be able to write `node.fetch("httpi://<publickey>/path")` with the public key embedded in the URL, matching the `httpi://` scheme defined in the protocol spec.

## Evidence

- `packages/iroh-http-shared/src/fetch.ts:51-62` — current signature: `(peer, input, init?)`
- `docs/protocol.md` — defines `httpi://<public-key>/path` as the canonical URL format
- Observed during live testing: the API feels awkward because you pass the peer ID separately from the path, unlike any other HTTP client

## Impact

The non-standard signature makes the API harder to learn and prevents patterns like storing URLs as strings, passing them around, or using existing HTTP tooling. It also breaks the project's own principle of web-standard compatibility.

## Remediation

1. Parse `httpi://` URLs to extract the public key from the host and the path from the rest
2. Accept both forms: `node.fetch("httpi://nodeId/path")` and the current `node.fetch(peer, "/path")` for backwards compat
3. Update TypeScript types and docs

## Acceptance criteria

1. `node.fetch("httpi://<base32-public-key>/hello")` works and returns a Response
2. `node.fetch(peer, "/hello")` still works (backward compat)
3. Fetch signature matches `(input: string | URL | Request, init?: IrohFetchInit) => Promise<Response>`
