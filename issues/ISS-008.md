# [ISS-008] Error taxonomy relies heavily on string matching

## Summary

Core error classification and the JS adapter fallback classifier both use substring/regex matching on error message strings rather than a structured error code taxonomy. This violates the stated engineering principle of a finite code-based taxonomy across the FFI boundary.

## Evidence

- `crates/iroh-http-core/src/lib.rs:140` — core classification based on message substrings
- `packages/iroh-http-shared/src/errors.ts:262` — JS fallback regex classifier still in use
- `docs/principles.md:187` — principle requires a finite code-based taxonomy across FFI

## Impact

Any change to an error message wording can silently break callers that depend on the classifier's output. The error surface is fragile and hard to test exhaustively.

## Remediation

1. Promote `core_error_to_json` structured code path as the default everywhere.
2. Minimize and phase out `classify_error_json` and the regex fallback.
3. Add explicit error-code conformance tests.

## Acceptance criteria

1. All main error pathways are code-first, not string-pattern based.
