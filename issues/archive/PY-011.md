# [PY-011] IrohNode.serve panics when called outside an entered Tokio runtime

## Summary

`IrohNode.serve()` calls into core `serve()` from a synchronous context without entering a Tokio runtime. Core `serve()` immediately calls `tokio::spawn`, which panics when no reactor is entered.

## Evidence

- `packages/iroh-http-py/src/lib.rs:779-795` — `serve()` invokes `iroh_http_core::serve(...)` directly from sync method.
- `crates/iroh-http-core/src/server.rs:554` — core `serve()` uses `tokio::spawn` immediately.
- `packages/iroh-http-py/tests/test_node.py:79` — `server.serve(handler)` call path that triggers panic in tests.

## Impact

Serving requests can crash at runtime with `there is no reactor running`, making the Python server API unreliable in normal async test/application flows.

## Remediation

1. Ensure `iroh_http_core::serve(...)` is called with an entered Tokio runtime (for example via `let _guard = rt.enter();` around the call).
2. Alternatively move serve startup fully into a runtime task where a reactor context is guaranteed.
3. Add regression tests that call `node.serve(...)` from pytest-asyncio and assert no panic.

## Acceptance criteria

1. `node.serve(handler)` no longer panics with missing reactor in pytest-asyncio tests.
2. Serve loop starts and accepts requests in Python tests (`test_serve_fetch_*` passes).
3. Runtime entry expectations are documented in code comments near the binding call.
