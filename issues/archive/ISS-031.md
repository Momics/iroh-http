# [ISS-031] Fetch cancellation tokens are allocated with `ep_idx=0` across adapters

## Summary

All platform adapters allocate fetch-cancel tokens with `alloc_fetch_token(0)` instead of endpoint-specific indices. This bypasses endpoint-scoped cleanup semantics used by `unregister_endpoint`.

## Evidence

- `packages/iroh-http-node/src/lib.rs:543` — Node allocates token with `alloc_fetch_token(0)`
- `packages/iroh-http-deno/src/dispatch.rs:334` — Deno allocates token with `alloc_fetch_token(0)`
- `packages/iroh-http-tauri/src/commands.rs:261` — Tauri allocates token with `alloc_fetch_token(0)`
- `crates/iroh-http-core/src/stream.rs:328` — `unregister_endpoint` retains by `ep_idx`

## Impact

Cancellation-token ownership is detached from endpoint lifecycle. Endpoint teardown cannot reliably clean orphaned tokens, increasing lifetime ambiguity and cleanup drift in abnormal/cancel-heavy flows.

## Remediation

1. Allocate fetch tokens with the real endpoint index.
2. Keep endpoint-scoped cleanup behavior consistent across all slab registries.
3. Add adapter-level conformance checks to prevent hardcoded `ep_idx=0` regression.

## Acceptance criteria

1. Fetch tokens are tagged with their owning endpoint in all adapters.
2. Endpoint unregister removes endpoint-owned fetch tokens deterministically.
