# [ISS-030] Core cancellation code `CANCELLED` is not mapped by JS classifier

## Summary

Core structured errors encode cancellation as `CANCELLED`, but shared TypeScript classification handles `ABORTED` instead. The mismatch prevents aborts from being surfaced as `IrohAbortError`.

## Evidence

- `crates/iroh-http-core/src/lib.rs:136` — `ErrorCode::Cancelled => "CANCELLED"`
- `packages/iroh-http-shared/src/errors.ts:253` — classifier maps `"ABORTED"` to `IrohAbortError`
- `packages/iroh-http-shared/src/errors.ts:257` — unknown codes fall back to generic `IrohError`

## Impact

Abort/cancel paths may lose semantic error typing in JS runtimes, forcing callers to handle generic errors and undermining cross-runtime error-code consistency.

## Remediation

1. Align cancellation code naming across core and JS (`CANCELLED` vs `ABORTED`).
2. Add explicit compatibility mapping so both legacy and canonical names map to `IrohAbortError`.
3. Add conformance coverage for structured cancellation errors.

## Acceptance criteria

1. A core cancellation error is classified as `IrohAbortError` in shared JS.
2. Error taxonomy docs and adapters use one canonical cancellation code.
