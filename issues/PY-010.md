# [PY-010] Unsafe raw pointer dereference in async path_changes iterator

## Summary

`IrohPathChanges.__anext__` stores `self` as a raw pointer and dereferences it after an `await`. If the Python object is dropped while the future is pending, the pointer can become dangling and trigger undefined behavior.

## Evidence

- `packages/iroh-http-py/src/lib.rs:567` — stores `self` as `*mut IrohPathChanges` encoded as `usize`.
- `packages/iroh-http-py/src/lib.rs:568` — async future is created and awaited.
- `packages/iroh-http-py/src/lib.rs:577` — raw pointer is unsafely dereferenced after await.
- `packages/iroh-http-py/src/lib.rs:578-579` — writes through the dereferenced pointer to mutate state.

## Impact

This is a memory-safety defect. In the presence of object drop or lifecycle races, behavior is undefined and can manifest as crashes or silent corruption.

## Remediation

1. Remove raw-pointer state mutation from `__anext__`.
2. Move mutable iterator state into shared ownership (for example `Arc<tokio::sync::Mutex<...>>`) captured by the future.
3. Keep all state updates within safe Rust and avoid `unsafe` pointer writes across await points.

## Acceptance criteria

1. `IrohPathChanges.__anext__` contains no raw pointer to `self` and no unsafe dereference across await points.
2. Iterator state transitions (`last_relay`, initialization flag) are handled via safe shared state.
3. Existing behavior remains intact: iterator yields on path changes and stops when the peer disconnects.
