# [ISS-003] Connection statistics missing from peerStats — no bandwidth, stream count, or latency data

## Summary

`node.peerStats(peerId)` and `node.peerInfo(peerId)` return address/relay info but lack connection-level statistics. The QUIC transport (iroh-quinn) tracks bandwidth usage, open stream count, RTT, and connection state, but this data is not surfaced through the FFI bridge.

## Evidence

- Observed during live testing: `peerStats` and `peerInfo` return mostly the same address info; no throughput, stream, or latency metrics
- `peerInfo` for self returns null (expected, since it's for remote peers — but this should be documented)
- `docs/features/observability.md` likely describes planned stats that aren't wired up yet

## Impact

Users cannot monitor connection health, measure bandwidth, or debug performance issues. This is important for production use cases like streaming media, where knowing whether you're relayed vs. direct and what the throughput is matters.

## Remediation

1. Investigate what iroh-quinn exposes via its `Connection` stats API (RTT, bytes sent/received, streams open, congestion window, etc.)
2. Wire these through the Rust core's dispatch layer to the FFI bridge
3. Return them as part of `peerStats()` or a new `connectionStats()` method
4. Document the difference between `peerInfo` (address discovery) and `peerStats` (connection metrics)

## Acceptance criteria

1. `peerStats(peerId)` includes at minimum: RTT, bytes sent/received, open QUIC stream count, relay vs. direct status
2. Documentation clarifies what each stat field means
3. `peerInfo` for self is documented as returning null (or returns useful own-node info)
