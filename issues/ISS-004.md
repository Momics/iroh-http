# [ISS-004] Trailer headers on incoming requests — verify req.trailers is accessible in serve handler

## Summary

The serve layer attaches a `trailers` property to incoming Request objects via `Object.defineProperty` (serve.ts:188-193), but it's unclear whether this works correctly end-to-end. The native `Request` class doesn't have a `trailers` property, so consumers need to know to access `(req as any).trailers` or cast to an extended type. This needs verification and proper typing.

## Evidence

- `packages/iroh-http-shared/src/serve.ts:188-193` — trailers attached via defineProperty
- `docs/features/trailer-headers.md` — trailer headers are a documented feature
- During testing, it was unclear whether trailer headers are actually readable on the server side

## Impact

If trailer headers aren't properly exposed, one of the advertised features doesn't work. Even if the mechanism works, the lack of TypeScript typing means users won't discover it.

## Remediation

1. Verify end-to-end: send a request with trailers from client, read them in server handler
2. Export an `IrohRequest` type (extending Request) that includes the `trailers` promise
3. Document how to read trailer headers in the serve handler
4. Add a test case

## Acceptance criteria

1. Server handler can `await req.trailers` and read trailer header values
2. TypeScript types expose `trailers` property without requiring casts
3. Example in docs/features/trailer-headers.md shows both send and receive
