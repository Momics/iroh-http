# [BUG-003] `withTimeout` timer leaks when a test errors mid-flight

## Summary

The inline `withTimeout` helper in Deno tests leaves a dangling `setTimeout` when the inner promise rejects before the timer fires. Deno's leak detector flags this as a resource leak.

## Evidence

Observed test output:
```
error: Leaks detected:
  - A timer was started in this test, but never completed.
  - An async operation to do a non blocking ffi call was started in this test, but never completed.
```

This is a consequence of BUG-001; once that is fixed, the leak typically disappears. If `withTimeout` is retained, it must clear the timer in a `finally` block.

## Impact

Deno's `--leak-detection` mode warns on every test that triggers BUG-001, obscuring other real leaks.

## Remediation

1. Fix BUG-001 first — that will eliminate most occurrences.
2. If `withTimeout` is kept, clear the timer in a `finally` block:
```ts
function withTimeout<T>(ms: number, fn: () => Promise<T>): Promise<T> {
  let id: ReturnType<typeof setTimeout>;
  const timer = new Promise<never>((_, reject) => {
    id = setTimeout(() => reject(new Error(`Timed out after ${ms}ms`)), ms);
  });
  return Promise.race([fn().finally(() => clearTimeout(id!)), timer]);
}
```

## Acceptance criteria

1. Deno tests run without timer-leak warnings.
