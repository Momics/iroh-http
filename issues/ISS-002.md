# [ISS-002] Duplex path ignores handler response status and always returns 101

## Summary

The server's upgrade path awaits the JS handler response head but then unconditionally sends `101 Switching Protocols`, discarding whatever status the handler returned. Handlers cannot reject or authorize duplex requests.

## Evidence

- `crates/iroh-http-core/src/server.rs:292` — upgrade path ignores handler status
- `crates/iroh-http-core/src/server.rs:355` — always emits 101 regardless of handler decision
- `packages/iroh-http-shared/src/serve.ts:227` — shared handler correctly returns `res.status` for bidi

## Impact

A handler returning `403 Forbidden` will still upgrade the connection to duplex. This is a potential authentication/authorization bypass for any application that expects non-101 to reject a duplex request.

## Remediation

1. In the server upgrade path, read and honor the handler's status and headers.
2. Only perform the upgrade pump when the status is `101` with valid `Upgrade` semantics.
3. Return non-101 responses as normal HTTP responses.

## Acceptance criteria

1. A duplex handler returning `403` sends `403` to the client and does not upgrade.
2. The existing valid duplex path still upgrades on `101`.
