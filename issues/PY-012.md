# [PY-012] node.closed can block forever after close completes

## Summary

`node.closed` uses `Notify::notified()` while `close()` signals via `notify_waiters()`. This is not a persistent closed-state signal, so waiting on `node.closed` after close completion can hang indefinitely.

## Evidence

- `packages/iroh-http-py/src/lib.rs:825-831` — `close()` calls `notify.notify_waiters()`.
- `packages/iroh-http-py/src/lib.rs:844-849` — `closed` awaits `notify.notified().await`.
- `packages/iroh-http-py/src/lib.rs:839` — docs promise `await node.closed` resolves once closed.

## Impact

Callers can deadlock or stall shutdown flows if they await `node.closed` after `close()` has already completed, violating the documented lifecycle contract.

## Remediation

1. Replace one-shot `Notify` semantics with persistent closed state (for example `watch<bool>` or `AtomicBool` + notify).
2. In `closed`, check closed state first and return immediately if already closed.
3. Add tests that await `node.closed` both before and after `await node.close()`.

## Acceptance criteria

1. `await node.closed` resolves if called before `close()` and also if called after `close()` has finished.
2. Lifecycle tests cover both orders and pass reliably.
3. Documentation for `node.closed` matches implemented semantics.
