# [ISS-028] Slab sweep restart is broken due to `OnceLock` shutdown handle

## Summary

The slab sweeper restart path is inconsistent: `start_slab_sweep()` creates a new shutdown `Notify` on restart, but `SWEEP_SHUTDOWN` is a `OnceLock`, so the new notify is never stored. Later `stop_slab_sweep()` signals the stale notify, not the active task.

## Evidence

- `crates/iroh-http-core/src/stream.rs:624` — `SWEEP_SHUTDOWN.set(shutdown.clone())` result is ignored
- `crates/iroh-http-core/src/stream.rs:645` — stop uses `SWEEP_SHUTDOWN.get()` (stale first notify after restart)
- `crates/iroh-http-core/src/stream.rs:649` — `SWEEP_STARTED` reset allows new starts even if current sweep task was not signaled

## Impact

After endpoint restart cycles, sweep tasks may become un-stoppable via `stop_slab_sweep()`, and duplicate sweepers can be started. This risks long-lived background task leaks and unstable registry-cleanup behavior.

## Remediation

1. Replace the `OnceLock<Notify>` shutdown slot with a replaceable cell (e.g., `Mutex<Option<Arc<Notify>>>`) or equivalent restart-safe mechanism.
2. Ensure `start_slab_sweep` and `stop_slab_sweep` always operate on the same active notify instance.

## Acceptance criteria

1. Start/stop/start/stop cycles always stop the currently running sweep task.
2. At most one sweep task is running process-wide across repeated endpoint lifecycle operations.
