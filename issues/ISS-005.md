# [ISS-005] First request often fails with "invalid endpoint handle" — needs graceful retry or better error

## Summary

The first fetch() call to a remote peer often fails with an "invalid endpoint handle" error. Subsequent calls succeed. This is likely a race condition where the remote peer's public key hasn't been discovered via DNS/relay yet when the first request fires. The error message is misleading — it sounds like an internal bug rather than a connectivity timing issue.

## Evidence

- Observed during live testing: first request at 3s interval failed, worked at 1s intervals (likely hit after discovery completed), then consistently worked
- The `paths` array on first successful request had only 1 entry (relay); subsequent requests had 2 (relay + direct IP)
- Error message "invalid endpoint handle" doesn't suggest the real cause

## Impact

Poor first-use experience. Users will think the library is broken when the first request fails. The error message doesn't help them understand what happened or what to do.

## Remediation

1. **Investigate root cause** — is this truly a DNS propagation delay, a QUIC handshake timeout, or a handle lifecycle issue?
2. **Better error message** — if the peer isn't found yet, the error should say "peer not found" or "connection timeout" rather than "invalid endpoint handle"
3. **Consider built-in retry** — optionally retry the first connection with backoff, or expose a `waitForPeer(peerId)` method
4. **Document the warmup** — explain that the first connection may take a few seconds while peer discovery completes

## Acceptance criteria

1. Error message clearly indicates "peer not reachable" rather than "invalid handle"
2. Documentation mentions the initial connection delay
3. Ideally: first fetch blocks until peer is discovered (with configurable timeout) rather than failing immediately
