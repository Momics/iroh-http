# [ISS-005] `maxConnectionsPerPeer` behavior mismatch — QUIC close vs HTTP 429

## Summary

When `maxConnectionsPerPeer` is exceeded, the server closes the QUIC connection. Documentation promises an HTTP `429 Too Many Requests` response.

## Evidence

- `crates/iroh-http-core/src/server.rs:517` — implementation closes connection when limit is exceeded
- `docs/features/server-limits.md:49` — docs claim `429` response behavior
- `docs/features/rate-limiting.md:19` — same `429` claim

## Impact

Clients receive a transport-level connection close instead of an HTTP error code. Callers relying on `429` for backoff logic will not behave as expected.

## Remediation

1. Decide on the intended semantics: transport-close or HTTP `429`.
2. Align code and docs to the chosen behavior.
3. Add an explicit integration test for the chosen behavior.

## Acceptance criteria

1. One canonical behavior is documented and tested end-to-end.
