# [PY-015] Python adapter contains protocol logic and wrapper classes beyond thin FFI shim

## Summary

The Python adapter's `src/lib.rs` (~700 lines) implements full wrapper classes (`IrohRequest`, `IrohResponse`, `IrohBidiStream`, `IrohSession`, `IrohBrowseSession`) with protocol logic, async iteration, and context managers — far exceeding the "thin shim" architecture.

## Evidence

- `packages/iroh-http-py/src/lib.rs` — `IrohBidiStream` with `write()`, `read()`, `close()` methods containing error guard logic (e.g., "PY-006: write() is not supported on a receive-only unidirectional stream")
- `packages/iroh-http-py/src/lib.rs` — `IrohBrowseSession` implementing `__aiter__` / `__anext__` for async iteration
- `packages/iroh-http-py/src/lib.rs` — `IrohSession` implementing `__aenter__` / `__aexit__` async context managers
- `docs/architecture.md:37-40` — "Thin shims. Translate platform types ↔ u64 handles. No logic. No state."

**Unlike JS/TS adapters:** The JS adapters delegate high-level API composition to `iroh-http-shared`. Python has no equivalent shared layer, so all this logic lives in the adapter itself.

## Impact

- Bug fixes to stream semantics (e.g., changing the "receive-only guard" logic) must be ported across Python and JS independently.
- Python's richer API surface (async context managers, async iteration, receive-only guards) creates behavioral differences that aren't documented as intentional platform deviations.
- Missing features in Python that exist in JS (e.g., `maxPooledConnections`, `poolIdleTimeoutMs` not exposed in `create_node`).

## Remediation

1. **Accept the deviation and document it:** Python's adapter is necessarily thicker than JS adapters because Python has no equivalent to `iroh-http-shared`. Add a doc comment at the top of `lib.rs` explaining that for Python, the adapter serves as both FFI shim and API composition layer.
2. **Extract protocol-level guards to core:** Move receive-only stream guards and similar checks from PY-006 into `iroh-http-core`. The Rust core should return a typed error when `send_chunk` is called on a read-only handle, rather than relying on each adapter to implement guards.
3. **Expose all NodeOptions fields:** Ensure `create_node` in Python accepts `max_pooled_connections` and `pool_idle_timeout_ms` to match JS adapter parity.

## Acceptance criteria

1. Protocol-level validations (e.g., write-on-read-only guard) exist in `iroh-http-core`, not in the Python adapter.
2. Python `create_node` accepts all the same configuration fields as JS `createNode`.
3. Either the adapter's thickness is documented as intentional, or logic is extracted to core.
