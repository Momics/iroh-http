# [ISS-029] `ServeHandle.finished` is wired to node close, not serve-loop termination

## Summary

The public `ServeHandle.finished` contract says it resolves when the serve loop terminates, but shared wiring passes a promise derived from node `closed` state. This mismatches lifecycle semantics for `stopServe(signal)` and fatal serve-loop exits.

## Evidence

- `packages/iroh-http-shared/src/serve.ts:79` — contract: resolves when serve loop terminates
- `packages/iroh-http-shared/src/index.ts:178` — `makeServe(..., closedPromise.then(() => {}), ...)`
- `packages/iroh-http-shared/src/index.ts:347` — `resolveClosed(...)` is called from node `close()`

## Impact

Callers awaiting `server.finished` can block until node close even when serve has already stopped, or observe incorrect lifecycle ordering. This can break shutdown orchestration and resource cleanup.

## Remediation

1. Introduce a dedicated serve-loop completion signal from raw adapters/core.
2. Wire `ServeHandle.finished` to serve-loop completion, not node-wide closed state.
3. Keep node `closed` and serve `finished` as distinct lifecycle promises.

## Acceptance criteria

1. Aborting `serve({ signal })` resolves `ServeHandle.finished` without requiring `node.close()`.
2. `node.closed` and `ServeHandle.finished` each reflect their documented lifecycle scope.
