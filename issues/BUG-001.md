# [BUG-001] Concurrent Deno responses are mis-routed to wrong callers

## Summary

Ten concurrent `client.fetch()` calls receive each other's response bodies. The Deno serve polling loop dispatches responses FIFO from the mpsc queue with no per-handle correlation, so `Promise.all` receives bodies out of order relative to the originating request.

## Evidence

Observed test failure:
```
error: AssertionError: Values are not equal: response 8 body mismatch
  Actual:   echo:/path6
  Expected: echo:/path8
```

- `packages/iroh-http-deno/src/adapter.ts` — `rawServe` polling loop has no per-handle correlation
- `packages/iroh-http-deno/src/dispatch.rs` — `next_request`, `serve_start` dispatch path

## Impact

In production, two concurrent Deno clients can silently receive each other's response bodies. This is a data-integrity correctness bug.

## Remediation

1. Re-examine whether mis-routing is in the serve dispatch path or in the client-side fetch queue.
2. Add per-fetch correlation logging with `--trace-leaks` to isolate the root cause.
3. Ensure each `rawFetch` awaits its own correlated response identified by `reqHandle`.

## Acceptance criteria

1. 100 concurrent `fetch()` calls each receive the correct response body.
